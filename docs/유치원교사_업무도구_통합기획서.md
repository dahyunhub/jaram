# 🌼 유치원 교사 업무 효율화 AI 서비스 — 통합 기획서

> 교사가 짧은 메모만 남기면, AI가 서류를 만들고, 기록은 **아이별로 자동 축적**되는 교사 전용 업무 도구

이 문서 하나로 어디서든(=Cowork) 바로 개발에 착수할 수 있도록 **기획 · 인터뷰 검증 · 화면 흐름 · 디자인 시스템 · 데이터 모델 · AI 설계 · 기술 스택**을 한곳에 모았다.

**최종 갱신:** 최초 기획 → 현직 교사 인터뷰 반영 → 화면 플로우/ERD → UI·UX 디자인까지 통합한 버전.

---

## 1. 프로젝트 개요

### 1.1 배경
유치원 현장에는 정말 중요하지 않은 행정 서류가 많고, AI 기술과 멀리 떨어져 있다. 교사들은 일지 작성에 챗봇형 AI를 쓰긴 하지만, 유치원 교사를 위한 전용 서비스는 없다. 일지를 한글 파일로 쓰다 보니 "누구를 기록했는지"가 남지 않아 따로 수첩에 수기로 적어야 했다.

### 1.2 핵심 가치
이 기획의 진짜 가치는 "AI가 일지를 써준다"가 아니다. 그건 이미 챗봇으로 하고 있다. 진짜 페인포인트는 **기록이 '문서 단위'로 쌓이지 '아이 단위'로 쌓이지 않는다는 것**이다.

한글 파일에 일지를 쓰면 그 기록은 날짜에 묶여 죽어버린다. "민준이가 최근 한 달간 어땠지?"에 답하려면 파일 30개를 열어야 한다. 이 구조를 뒤집는 게 서비스의 핵심이다.

**서류 작성(출력물)과 아동 관찰 데이터(자산)를 분리하되, 한 번의 입력으로 둘 다 해결되는 구조.**

### 1.3 범위 (중요)
- **포함**: 오로지 교사만 사용하는, 교사의 업무를 줄여주는 도구. 서류를 줄이고, 아이를 교육·케어하는 데 도움을 준다.
- **제외**: 학부모와의 소통 창구가 **아니다.**

## 2. 현직 교사 인터뷰 결과 (기획의 근거)

현직 동료 교사에게 직접 인터뷰한 결과로, MVP의 방향을 확정한 핵심 근거다. **README·면접에서 "현직 교사 인터뷰 기반으로 기능을 검증했다"**는 한 줄로 쓸 수 있다.

### 2.1 입력 부담: 3~5줄은 괜찮다 — 단, "구조"가 있어야 한다
교사는 메모를 쓸 때 자연스럽게 세 항목으로 나눠 말했다: **① 어떤 놀이를 했는지 ② 놀이 중 어떤 의사소통/상호작용을 했는지 ③ 수업 태도는 어떤지.**
→ 메모를 자유 입력 한 칸으로만 두지 말고, 이 세 항목을 **가벼운 입력 가이드**로 제시한다. 빈 칸 공포를 줄이면서, 동시에 AI가 더 정확히 쓸 수 있는 구조화된 데이터가 된다. (입력 10초 원칙은 유지, 항목은 전부 선택)

### 2.2 경쟁자(ChatGPT)의 약점을 교사가 직접 짚어줬다 — 서비스의 존재 이유
> "GPT한테 시키면 놀이가 한정적이다. 상호작용 과정이나 교사의 지원이 같이 써지면 좋겠다."

범용 ChatGPT는 유아교육 전문 프롬프트가 없어 매번 납작한 놀이만 뱉는다. 이 서비스는 **누리과정 기반 도메인 특화 프롬프트를 내장**해 버튼 하나로:
- 놀이를 다양하게 제안 (같은 주제라도 신체·언어·미술·역할놀이 등으로 확장)
- **또래 상호작용 과정**을 서술에 포함
- **교사의 지원/발문**을 자동으로 함께 작성

→ 면접 포인트: "범용 AI 대비 도메인 특화 프롬프트로 차별화했다."

### 2.3 우선순위가 명확해졌다 — MVP의 타깃 서류
교사가 "제일 먼저 해결됐으면 하는 서류"로 콕 집은 것:
1. **상담일지용 개별 관찰일지** → 출력물 A와 직결
2. **주안 작성 시 놀이사항 및 평가** → 출력물 B와 직결

## 3. MVP 범위

1인 프로젝트에서 가장 흔한 실패는 기능 욕심이다. 출석부·준비물 체크·알림은 모두 v2로 미루고, MVP는 아래 흐름 하나에 집중한다.

### 3.1 핵심 기능
1. **빠른 메모 입력** — 하루 중 틈틈이 한두 줄~몇 줄 입력. 아이 이름 태그가 핵심(여기서 아이–기록 연결이 생긴다). 입력은 〈놀이 / 의사소통·상호작용 / 수업태도〉 3개 항목 가이드로 부담 없이.
2. **AI 일지/분석 생성** — 그날 쌓인 메모를 기반으로 일지·분석 초안 생성. 교사가 검토·수정 후 확정. 누리과정 영역(신체운동·건강, 의사소통, 사회관계, 예술경험, 자연탐구)에 맞춰 분류.
3. **아이별 타임라인 페이지** — 아이를 클릭하면 지금까지의 모든 관찰 기록이 시간순으로 보임. 학부모 상담·발달평가에 활용.

### 3.2 두 가지 출력물 (인터뷰로 구체화)
입력 소스는 똑같은 "빠른 메모". 한 번 입력하면 두 방향으로 동시에 쓰인다.

| 출력물 | 묶는 기준 | 결과물 | 용도 |
|--------|-----------|--------|------|
| **A · 개별 관찰일지** | 그 아이의 메모만 | 한 아이에 대한 관찰 서술 | 상담일지, 발달평가 |
| **B · 일지 / 주간 놀이 기록·평가** | 그날(또는 한 주) 반 전체 메모 | 놀이 흐름 + 또래 상호작용 + 교사 지원이 포함된 서술 | 매일 보육일지 / 주안 |

> **결정 필요(작은 사항):** B를 MVP에서 "하루 단위 보육일지"로 먼저 낼지, "주간(주안)"으로 낼지. 두 형태 모두 같은 메모 풀에서 기간만 바꿔 묶으면 되므로, **MVP는 하루 단위로 시작 → 기간 파라미터만 늘려 주간 지원**하는 방향을 권장.

### 3.3 v2 이후로 미루는 것
- 출석 관리, 준비물 체크, 알림
- 아이 명단 검색 (MVP는 가나다순 정렬로 충분)
- 일지 버전 관리 (MVP는 덮어쓰기)

## 4. 사용자 여정

마찰이 생기는 두 지점(낮 메모 입력, 밤 재분석)에 설계 역량을 집중한다.

| 단계 | 행동 | 감정 | 설계 기회 |
|------|------|------|-----------|
| **낮 · 틈틈이** (수업 중) | 아이 탭 + 한 줄 메모를 여러 번 | 분주함 ⚠ | 입력 10초 이내, 오프라인 대비 |
| **밤 · 일지 1차** (개인 업무 시간) | 분석 페이지에서 오늘 메모 확인 후 AI 분석 | 안도 | DB 먼저 저장해 안심 제공 |
| **밤 · 재분석** (깜빡한 메모) | 메모 창에서 추가 입력 → 다시 분석 | 번거로움 ⚠ | 시각 비교로 배너·new 뱃지 자동화 |
| **월말 · 자동** (교사 개입 없음) | 아무것도 안 함, 결과만 쌓임 | 편안함 | 메모 없는 아이 건너뛰기 |
| **상담 · 활용** (학부모 면담) | 아이 페이지에서 누적 평가 열람 | 자신감 | 평가를 시간순 누적 보관 |

**핵심 통찰:** 월말 자동 분석은 교사가 아무것도 안 해도 가치가 쌓이는 유일한 구간이다. 챗봇형 AI는 매번 시켜야 일하지만, 이 서비스는 가만히 있어도 학기말 자산이 쌓인다. 이것이 경쟁 서비스와의 진짜 차별점이다.

## 5. 화면 흐름

### 5.1 전체 구조
```
로그인 · 반 선택
      │
      ▼
   홈 (오늘의 메모)
   ├──────────────┐
   ▼              ▼
① 빠른 메모 입력   ② AI 일지 생성·검토
   │ (메모 저장 시          │ (확정 일지도
   │  아이별 자동 축적)      │  기록에 반영)
   └──────┬───────────────┘
          ▼
③ 아이별 타임라인 페이지 → AI 발달 요약 (v1.5)
          ▼
   아이 목록 (반 전체)
```

### 5.2 화면별 상세 명세 (각 화면 모바일·데스크톱 두 버전)

**화면 0 — 로그인 · 반 선택**
- 따뜻한 환영 톤. 로고 + 한 줄 카피("오늘도 우리 반, 함께해요" 같은).
- 로그인 후 담당 반 카드 선택(반 이름 + 아이 수). 작년/올해 반 구분 가능.

**화면 1 — 홈 (오늘의 메모)**
- 상단: 인사("○○ 선생님, 좋은 하루예요 ☀️") + 오늘 날짜 + 반.
- 가장 눈에 띄는 곳에 '+ 빠른 메모' 진입(모바일은 플로팅 버튼).
- 오늘 쌓인 메모 피드(아이 이름 + 한 줄 + 영역 칩 + 시각).
- 눈에 띄는 'AI 일지 쓰기' 큰 버튼/카드.

**화면 2 — 빠른 메모 입력 ⭐(가장 자주 쓰는 화면)**
- 흐름: **아이 선택 → 내용 입력 → 저장**, 위에서 아래로 딱 세 동작.
- 아이 선택: 가나다순 명단(아바타/이니셜 원형 칩). 탭하면 선택.
- 입력: 자유 입력 + **빈 칸 공포를 줄이는 항목 가이드** 〈놀이 / 의사소통·상호작용 / 수업태도〉(현직 교사 인터뷰 기반, 전부 선택사항, 한 줄씩 톡톡).
- 누리과정 **영역 태그는 AI 자동 추천 → 교사는 확인만** (입력 부담 0).
- 저장 토스트: **"민준이 페이지에 저장됐어요 ✓"** (아이별로 쌓인다는 체감).

**화면 3 — AI 일지 생성·검토 (상태 3개 + 재분석)**
AI API를 호출하므로 로딩·성공·에러 상태가 전부 필요. 프론트에서 상태 머신(`loading → success` / `loading → error`)으로 관리.
- **(a) 생성 중:** API가 10~20초 걸림. 빈 화면 금지 — "오늘 메모를 모아 일지를 쓰고 있어요. 잠시만요(약 15초)"처럼 **무엇을·얼마나** 안내. 귀여운 로딩 애니메이션(새싹·햇살).
- **(b) 초안 검토:** 누리과정 영역별로 분류된 카드. 인라인 수정 가능. 하단 [수정] [다시 생성] [확정].
- **(c) 에러 ⭐:** 가장 중요한 메시지 — **"작성하신 메모는 그대로 저장돼 있어요."** 색은 빨강 아닌 살구코랄. [다시 시도] [메모만 보기].
- **분석 메뉴 2종이 UI에서 구분:** A·한 아이 분석(개인 관찰 평가 → 상담·발달평가) / B·오늘 전체 분석(보육일지 → 매일 제출 서류).

**화면 3-1 — 재분석 흐름**
한 페이지를 위→아래 3구역으로:
- **상단:** 안내 배너(새 메모 있을 때만) — "새로 입력된 메모가 있어요. 분석을 한 번 더 해볼까요?"
- **중간:** 오늘 메모 목록(방금 추가분은 `NEW` 뱃지).
- **하단:** 이전 분석 결과(회색 톤으로 죽여 '이미 만든 것' 표현 + 생성 시각).
- **버튼:** 분석이 한 번이라도 있으면 [다시 분석하기], 처음이면 [AI 분석].
- [다시 분석하기] → "오늘 분석 내용이 이미 있어요. 다시 만들까요?" 모달 → [다시 분석]/[취소]. 다시 분석 시 기존 분석에 덮어쓰기.
- **배너 판정(MVP):** `메모.생성시각 > 분석.생성시각`인 메모가 하나라도 있으면 배너 + NEW 뱃지. 덮어쓰기 후 분석 시각이 갱신되면 배너가 자동으로 사라진다.

**화면 4 — 아이별 타임라인 ⭐(서비스의 진짜 가치)**
- 상단: AI 발달 요약 블록(v1.5, 학기말 평가용, 부드러운 카드).
- 영역 필터 칩(신체운동/의사소통/사회관계/예술경험/자연탐구) — 색 코딩 적용(§6.2).
- 시간순 관찰 기록 타임라인(각 카드 좌측에 영역 색띠, 날짜·내용·영역 칩).

**화면 5 — 아이 목록 (반 전체)**
- 가나다순 그리드/리스트. 아바타(이니셜 원형) + 이름 + 최근 메모 수 뱃지. 탭하면 화면 4로.

## 6. UI · UX 디자인 시스템

전체 디자인은 **Claude Design**으로 진행. 컨셉은 **"다정한 전문성(warm professionalism)"** — 둥글고 귀엽되, 상담 자리에서 학부모 앞에 펼쳐도 괜찮을 만큼 정돈됨.

### 6.1 디자인 원칙 (5)
1. **다정한 전문성** — 둥글고 귀엽되 유치하지 않게.
2. **한눈에 읽히는 메뉴** — 글자+아이콘+색으로 즉시 구분.
3. **입력은 가볍게, 부담은 0** — 빈 칸 공포 제거, 선택 입력은 AI 추천을 '확인'만.
4. **자산은 절대 안 사라진다는 안심** — 메모(자산)와 일지(출력물) 분리. 에러가 나도 "메모는 그대로 있어요"가 화면에 드러나야.
5. **따뜻하고 밝은 톤** — 어둡거나 칙칙한 색 금지. 봄·햇살·파스텔.

### 6.2 컬러 토큰 (봄 파스텔 + 노랑)
**브랜드/포인트 (파스텔 옐로우)**
- `--brand-100` `#FFF6DC` / `--brand-300` `#FFE08A` / `--brand-500` `#FFD45E`(메인) / `--brand-700` `#F5B940`(진한 포인트, 위에 진한 텍스트)

**배경/표면 (따뜻한 아이보리)**
- `--bg` `#FFFCF5` / `--surface` `#FFFFFF` / `--surface-soft` `#FFFBF2`

**텍스트 (순흑 대신 따뜻한 브라운그레이)**
- `--text` `#43392E`(본문) / `--text-sub` `#8C8478` / `--text-faint` `#B7AFA2`

**상태색 (전부 부드럽게)**

일과 중에는 폰, 일지 작성 시에는 PC를 주로 쓴다 → 반응형 웹.

- **데스크톱(≥768px):** 좌측 고정 사이드바. 일지 화면은 **2단(좌: 오늘 메모 목록 / 우: AI 결과)**. 검토·편집 편의.
- 같은 화면이라도 맥락별로 우선순위가 다름(모바일=입력 속도, 데스크톱=검토 편의).

### 6.6 글로벌 내비게이션
- 항목: **홈 · 아이들 · 일지/분석 · 마이**. 모바일=하단 탭바, 데스크톱=좌측 사이드바. 현재 위치는 브랜드 옐로우 강조.
- 상단(또는 사이드바 상단)에 현재 반(예: 햇살반) · 날짜 표시. 반 전환 가능.

### 6.7 톤 & 마이크로카피 / 접근성
- 존댓말, 따뜻하고 다정하되 과하지 않게. '선생님' 호칭.
- 예) 저장: "○○이 페이지에 저장됐어요 ✓" / 에러: "메모는 안전하게 있어요. 다시 해볼까요?" / 빈 타임라인: "아직 기록이 없어요. 오늘 첫 메모를 남겨볼까요?"
- **접근성:** 노랑 위 흰 글자 금지. 탭 영역 최소 44×44px. 본문 최소 16px. 파스텔이라도 텍스트·아이콘 대비 충분히.

## 7. AI 분석 설계 (비용·품질 핵심)

### 7.1 분석 타이밍 — 모아서 한 번만
메모마다 AI를 호출하면 비용이 메모 수만큼 늘고 맥락도 빈약하다. 그래서:
> 낮에는 메모를 **DB에만** 저장하고, 밤에 일지 쓸 때 **하루치 메모를 모아 한 번만** 분석한다.

모아 보면 AI가 "민준이는 오늘 또래관계에서 긍정적 변화를 여러 번 보였다"처럼 **패턴**을 읽는다. 메모를 하나씩 분석하면 안 나오는 문장이다.

### 7.2 분석 메뉴 두 가지

| 메뉴 | 묶는 기준 | 결과물 | 용도 |
|------|-----------|--------|------|
| **A · 한 아이 분석** | 그 아이의 메모만 | 개인 관찰 평가 | 학부모 상담, 발달평가 |
| **B · 오늘 전체 분석** | 그날 전체 메모(여러 아이) | 하루 보육일지 | 매일 제출하는 행정 서류 |

### 7.3 메뉴 A — 수동 + 월말 자동
- **수동:** 교사가 원할 때 그날까지의 분석 결과 도출(예: 갑자기 상담이 잡힘).
- **월말 자동:** 매월 말 스케줄러(`@Scheduled`)가 반 전체 아이를 순회하며 개별 분석. 교사가 아무것도 안 해도 발달 기록이 쌓이는 안전망.
  - 20명이면 AI를 20번 호출 → 한꺼번에 쏘지 말고 **순차 처리**.
  - 그달 메모 없는 아이는 빈 분석 만들지 말고 **건너뜀**.

### 7.4 메뉴 B — 일지 작성 & 재분석 흐름
1. '일지 쓰기' → AI 분석 페이지로 이동.
2. 페이지에 오늘 메모 내역 전부 표시.
3. [AI 분석] → 기존 분석이 없으면 하루치 메모 전체 기반으로 분석.
4. (재분석) 깜빡한 일이 생각남 → 메인의 **메모 창에서 평소처럼 입력**.
5. 다시 분석 페이지로 오면 새 메모가 목록에 포함됨 → §5.2 화면 3-1의 배너/뱃지/덮어쓰기 규칙 적용.

### 7.5 프롬프트 설계 요구사항 (인터뷰 직격)
범용 GPT 대비 차별점을 만드는 핵심. A·B 프롬프트 모두 다음을 강제한다.
- **누리과정 5영역 기반**으로 서술·분류.
- 같은 주제라도 **다양한 놀이**를 제안(신체·언어·미술·역할 등 확장).
- **또래 상호작용 과정**을 서술에 포함.
- **교사의 지원/발문**을 함께 작성.
- 메모의 3항목(놀이 / 의사소통·상호작용 / 수업태도)을 입력 신호로 활용.

## 8. 데이터 모델 (ERD)

### 8.1 관계 요약
```
TEACHER ──< CLASSROOM ──< CHILD ──< MEMO
   │            │                     │
   └──< MEMO    └──< DAILY_JOURNAL    └──< CHILD_REPORT

                     └──< JOURNAL_MEMO_LINK >── MEMO
```

## 10. 기술 스택

### 10.1 결정 원칙
목표는 **배포까지 완성해 포트폴리오로 만드는 것**. 따라서 **새로 배울 것을 최소화**한다. 이번의 새 도전은 **Docker 하나**로 좁힌다.

### 10.2 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 언어·프레임워크(백엔드) | **Java + Spring Boot** | 이미 학습 중. 새 도전을 Docker로 좁혀 완주율↑. 채용 수요 큼 |
| DB | **MySQL** | 학습 중, 무료, 가벼운 Docker 이미지. 정형 데이터라 관계형 DB 적합 |
| 빌드 도구 | **Gradle** | 설정 간결(의존성 1줄), 빌드 캐시로 빠름, 최신 자료 다수, Spring 공식 생성기 기본 |
| AI | **Claude / OpenAI API** | Spring에서 호출. 도메인 특화 프롬프트가 차별점 |
| 디자인 | **Claude Design** | §6 디자인 시스템 기반 화면 제작 |
| 인프라 (새 도전) | **Docker + docker-compose** | 백엔드+MySQL 컨테이너화 → 배포. 이력서 "Docker 기반 배포" 한 줄 |

> **프론트엔드 구현 프레임워크:** 디자인은 Claude Design으로 진행. 구현 프레임워크는 추후 확정(이전 논의에서 Vue 3 Composition API가 후보로 언급됨 — 새로 배울 것을 늘리지 않는 선택 권장).
