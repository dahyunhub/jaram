# 유치원 교사 업무 효율화 AI 서비스 — 프로젝트 기획서

> 교사가 짧은 메모만 남기면, AI가 서류를 만들고, 기록은 아이별로 자동 축적되는 **교사 전용 업무 도구**

이 문서 하나로 어디서든 프로젝트에 착수할 수 있도록 기획·화면 흐름·데이터 모델·기술 스택을 정리했다.

---

## 1. 프로젝트 개요

### 1.1 배경
유치원 현장에는 정말 중요하지 않은 행정 서류가 많고, AI 기술과 멀리 떨어져 있다. 교사들은 일지 작성에 챗봇형 AI를 쓰긴 하지만, 유치원 교사를 위한 전용 서비스는 없다. 일지를 한글 파일로 쓰다 보니 "누구를 기록했는지"가 남지 않아 따로 수첩에 수기로 적어야 했다.

### 1.2 핵심 가치
이 기획의 진짜 가치는 "AI가 일지를 써준다"가 아니다. 그건 이미 챗봇으로 하고 있다. 진짜 페인포인트는 **기록이 '문서 단위'로 쌓이지 '아이 단위'로 쌓이지 않는다는 것**이다.

한글 파일에 일지를 쓰면 그 기록은 날짜에 묶여 죽어버린다. "민준이가 최근 한 달간 어땠지?"에 답하려면 파일 30개를 열어야 한다. 이 구조를 뒤집는 게 서비스의 핵심이다.

**서류 작성(출력물)과 아동 관찰 데이터(자산)를 분리하되, 한 번의 입력으로 둘 다 해결되는 구조.**

### 1.3 범위 (중요)
- **포함**: 오로지 교사만 사용하는, 교사의 업무를 줄여주는 도구. 서류를 줄이고, 아이를 교육·케어하는 데 도움을 준다.
- **제외**: 학부모와의 소통 창구가 **아니다.**

## 2. MVP 범위

1인 프로젝트에서 가장 흔한 실패는 기능 욕심이다. 출석부·준비물 체크·알림은 모두 v2로 미루고, MVP는 아래 흐름 하나에 집중한다.

### 2.1 핵심 기능
1. **빠른 메모 입력** — 하루 중 틈틈이 "민준 · 블록놀이 중 양보 갈등, 스스로 해결함" 수준의 한두 줄 입력. 아이 이름을 태그하는 게 핵심(여기서 아이–기록 연결이 생긴다).
2. **AI 일지/분석 생성** — 그날 쌓인 메모를 기반으로 일지·분석 초안 생성. 교사가 검토·수정 후 확정. 누리과정 영역(신체운동·건강, 의사소통, 사회관계 등)에 맞춰 분류.
3. **아이별 타임라인 페이지** — 아이를 클릭하면 지금까지의 모든 관찰 기록이 시간순으로 보임. 학부모 상담·발달평가에 활용.

### 2.2 v2 이후로 미루는 것
- 출석 관리, 준비물 체크, 알림
- 아이 명단 검색 (MVP는 가나다순 정렬로 충분)
- 일지 버전 관리 (MVP는 덮어쓰기)

## 3. 사용자 여정

교사의 하루를 따라가며, 각 단계의 행동·접점·감정을 정리했다. 마찰이 생기는 두 지점(낮 메모 입력, 밤 재분석)에 설계 역량을 집중한다.

| 단계 | 행동 | 감정 | 설계 기회 |
|------|------|------|-----------|
| **낮 · 틈틈이** (수업 중) | 아이 탭 + 한 줄 메모를 여러 번 | 분주함 ⚠ | 입력 10초 이내, 오프라인 대비 |
| **밤 · 일지 1차** (개인 업무 시간) | 분석 페이지에서 오늘 메모 확인 후 AI 분석 | 안도 | DB 먼저 저장해 안심 제공 |
| **밤 · 재분석** (깜빡한 메모) | 메모 창에서 추가 입력 → 다시 분석 | 번거로움 ⚠ | 시각 비교로 배너·new 뱃지 자동화 |
| **월말 · 자동** (교사 개입 없음) | 아무것도 안 함, 결과만 쌓임 | 편안함 | 메모 없는 아이 건너뛰기 |
| **상담 · 활용** (학부모 면담) | 아이 페이지에서 누적 평가 열람 | 자신감 | 평가를 시간순 누적 보관 |

**핵심 통찰**: 월말 자동 분석은 교사가 아무것도 안 해도 가치가 쌓이는 유일한 구간이다. 챗봇형 AI는 매번 시켜야 일하지만, 이 서비스는 가만히 있어도 학기말 자산이 쌓인다. 이것이 경쟁 서비스와의 진짜 차별점이다.

## 4. 화면 흐름

### 4.1 전체 구조
```
로그인 · 반 선택
      │
      ▼
   홈 (오늘의 메모)
   ├──────────────┐
   ▼              ▼
① 빠른 메모 입력   ② AI 일지 생성·검토
   │  (메모 저장 시         │ (확정 일지도
   │   아이별 자동 축적)     │  기록에 반영)
   └──────┬───────────────┘
          ▼
③ 아이별 타임라인 페이지 → AI 발달 요약 (v1.5)
          ▼
   아이 목록 (반 전체)
```

### 4.2 화면 ① 빠른 메모 입력
- 위에서 아래로 딱 세 동작: **아이 이름 탭 → 한 줄 입력 → 저장**
- 아이 선택은 명단 리스트(세로 스크롤)에서 탭
- 누리과정 영역 태그는 선택사항. AI가 자동 추천한 걸 교사가 확인만 하게 해 입력 부담을 늘리지 않음
- 저장하면 아이 페이지에 자동 기록된다는 안내

### 4.3 화면 ② AI 일지 생성·검토 (3가지 상태)
AI API를 호출하므로 로딩·에러 상태가 반드시 필요하다. 프론트에서 상태 머신(`loading → success` / `loading → error`)으로 관리.

1. **생성 중**: API 호출이 10~20초 걸리므로 "무엇을 하고 있고 얼마나 걸리는지" 명시. 빈 화면이면 교사가 멈춘 줄 알고 뒤로가기를 누름.
2. **초안 검토**: 누리과정 영역별로 분류된 초안. 교사가 검토·수정 후 [수정] [다시 생성] [확정].
3. **에러**: 가장 중요한 메시지 — **"작성하신 메모는 그대로 저장돼 있어요."** 메모(자산)와 일지(출력물)를 분리한 설계가 UX로 드러나는 지점. [다시 시도] [메모만 보기].

### 4.4 화면 ③ 아이별 타임라인
- 상단: AI 발달 요약 블록 (v1.5, 학기말 평가용)
- 영역 필터(사회관계/신체운동 등)
- 시간순 관찰 기록 타임라인

## 5. AI 분석 설계 (비용·품질 핵심)

### 5.1 분석 타이밍 — 모아서 한 번만
메모를 쓸 때마다 AI를 호출하면 비용이 메모 수만큼 늘고, 메모 한 줄만으론 맥락이 빈약하다. 그래서:

> 낮에는 메모를 **DB에만** 저장하고, 밤에 일지 쓸 때 **하루치 메모를 모아 한 번만** 분석한다.

모아서 보면 AI가 "민준이는 오늘 또래관계에서 긍정적 변화를 여러 번 보였다"처럼 **패턴**을 읽어낼 수 있다. 메모를 하나씩 분석하면 절대 안 나오는 문장이다.

### 5.2 분석 메뉴 두 가지
교사가 쓰는 서류가 두 종류라서 분석도 둘로 나뉜다.

| 메뉴 | 묶는 기준 | 결과물 | 용도 |
|------|-----------|--------|------|
| **A · 한 아이 분석** | 그 아이의 메모만 | 개인 관찰 평가 | 학부모 상담, 발달평가 |
| **B · 오늘 전체 분석** | 그날 전체 메모(여러 아이) | 하루 보육일지 | 매일 제출하는 행정 서류 |

### 5.3 메뉴 A — 수동 + 월말 자동
- **수동**: 교사가 원할 때 그날 분석 결과를 도출 (예: 갑자기 상담이 잡힘)
- **월말 자동**: 매월 말 스케줄러(`@Scheduled`)가 반 전체 아이를 순회하며 개별 분석. 교사가 아무것도 안 해도 발달 기록이 쌓이는 안전망.
  - 주의: 20명이면 AI를 20번 호출 → 한꺼번에 쏘지 말고 순차 처리
  - 그달 메모가 없는 아이는 빈 분석을 만들지 말고 건너뜀

### 5.4 메뉴 B — 일지 작성 & 재분석 흐름
1. '일지 쓰기' 메뉴 → AI 분석 페이지로 이동
2. 페이지에 오늘 교사가 메모한 내역이 전부 표시됨
3. [AI 분석] 클릭 → 기존 분석이 없으면 하루치 메모 전체 기반으로 분석
4. **(재분석 시나리오)** 분석 후 깜빡한 일이 생각남 → 메인의 메모 창에서 평소처럼 입력
5. 다시 분석 페이지로 오면 새 메모가 목록에 포함됨

**"이미 분석한 경우" 화면 구성** — 한 페이지를 위에서 아래로 세 구역으로 쌓는다:
- **상단**: 안내 배너 (새 메모 있을 때만) — "새로 입력된 메모가 있어요. 분석을 한 번 더 해볼까요?"
- **중간**: 오늘 메모 목록 (방금 추가한 메모는 `new` 뱃지로 구분)
- **하단**: 이전 분석 결과 (회색 톤으로 죽여서 "이미 만든 것"임을 표현, 생성 시각 표시)
- **버튼**: 분석이 한 번이라도 있으면 [다시 분석하기], 처음이면 [AI 분석]

[다시 분석하기] → "오늘 분석 내용이 이미 있어요. 다시 만들까요?" 모달 → [다시 분석] / [취소]. 다시 분석 시 기존 분석에 덮어쓰기.

**배너 판정 (MVP)**: `메모.생성시각 > 분석.생성시각`인 메모가 하나라도 있으면 배너 + new 뱃지. 덮어쓰기 후 분석 시각이 갱신되면 배너가 자동으로 사라진다.

## 6. 데이터 모델 (ERD)

### 6.1 관계 요약
```
TEACHER ──< CLASSROOM ──< CHILD ──< MEMO
   │            │                     │
   └──< MEMO    └──< DAILY_JOURNAL    └──< CHILD_REPORT
                     │
                     └──< JOURNAL_MEMO_LINK >── MEMO
```
- 교사 1 : N 반
- 반 1 : N 아이
- 아이 1 : N 메모
- 일지 N : M 메모 (`JOURNAL_MEMO_LINK` 다리 테이블)
- 아이 1 : N 개인평가

### 6.2 테이블 명세

**TEACHER** (교사)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| email | string | 로그인 ID |
| password_hash | string | 비밀번호 해시 |
| name | string | |
| created_at | timestamp | |

**CLASSROOM** (반)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| teacher_id | bigint FK | |
| name | string | 반 이름 (예: 햇살반) |
| year | int | 학년도 — 작년 햇살반/올해 햇살반 구분 |
| created_at | timestamp | |

**CHILD** (아이)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| classroom_id | bigint FK | |
| name | string | 실명 (DB에만 저장) |
| birth_date | date | |
| token_alias | string | **비식별화 토큰** — AI API에 보낼 가명("아이A") |
| deleted_at | timestamp NULL | **soft delete** — 관찰 기록은 자산이므로 실제 삭제 안 함 |

**MEMO** (빠른 메모 · 서비스의 심장)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| child_id | bigint FK | 아이 기준 조회(타임라인)용 |
| teacher_id | bigint FK | 교사 기준 조회(메모 입력)용 |
| content | text | 메모 내용 |
| curriculum_area | string | 누리과정 영역 (AI 분류 또는 교사 태그) |
| created_at | timestamp | 배너 판정의 기준 시각 |
| deleted_at | timestamp NULL | soft delete |

> 인덱스: `child_id`, `created_at` 둘 다 — 메모 입력(교사+오늘)과 타임라인(아이+전체기간) 두 화면을 한 테이블이 먹인다.

**DAILY_JOURNAL** (보육일지 · 메뉴 B)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| teacher_id | bigint FK | |
| classroom_id | bigint FK | |
| journal_date | date | |
| ai_content | text | AI 생성 일지 본문 |
| version | int | MVP는 항상 1(덮어쓰기), v2에서 버전 관리 |
| created_at | timestamp | 배너 판정 기준 |

> 유니크 제약: `(teacher_id, classroom_id, journal_date)` — "하루에 일지 하나" 규칙을 DB가 강제

**JOURNAL_MEMO_LINK** (일지–메모 다리)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| journal_id | bigint FK | |
| memo_id | bigint FK | 어떤 메모가 어떤 일지에 반영됐는지 추적 |

**CHILD_REPORT** (개인 평가 · 메뉴 A)
| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | bigint PK | |
| child_id | bigint FK | |
| report_type | string | 'manual'(수동) / 'monthly'(월말 자동) |
| ai_content | text | AI 분석 결과 |
| period_start | date | 분석 대상 기간 시작 |
| period_end | date | 분석 대상 기간 끝 |
| created_at | timestamp | 시간순 누적 보관 (6월 평가, 7월 평가가 각각 레코드) |

## 7. 기술 스택

### 7.1 결정 원칙
이 프로젝트의 목표는 **배포까지 완성해 포트폴리오로 만드는 것**. 따라서 **새로 배울 것을 최소화**한다. 새 기술을 추가할 때마다 완주 확률이 떨어진다. 이번의 새 도전은 **Docker 하나**로 좁힌다.

### 7.2 스택
| 영역 | 선택 | 이유 |
|------|------|------|
| 언어·프레임워크 | **Java + Spring Boot** | 이미 학습 중. 새 도전을 Docker로 좁혀 완주율↑. 채용 수요 큼 |
| DB | **MySQL** | 학습 중, 무료, 가벼운 Docker 이미지. ERD가 정형 데이터라 관계형 DB가 적합 |
| 빌드 도구 | **Gradle** | 설정 간결(의존성 1줄), 빌드 캐시로 빠름, 최신 자료 다수가 Gradle 기준 |
| 인프라 (새 도전) | **Docker + docker-compose** | 백엔드+MySQL 컨테이너화 → 배포. "Docker 기반 배포" 이력서 한 줄 |

### 7.3 빌드 도구 보충 — Gradle
- **Gradle**: `build.gradle`, 간결(의존성 1줄), 빌드 캐시로 빠름, 최신 튜토리얼 다수. Spring 공식 생성기 기본 추천

## 8. 개인정보 보호 설계 (어필 포인트)

아동 관찰 기록은 민감정보다. 외부 AI API로 전송하는 순간 개인정보 이슈가 생긴다.

- DB에는 실명을 저장하되, AI API 호출 시 이름을 **가명 토큰(`token_alias`, "아이A")으로 치환**해 보내고 응답에서 복원
- README·면접 설명: **"도메인 특성상 아동 개인정보 보호를 위해 비식별화 레이어를 설계했다"**
- → 유아교육 경력 + 기술 설계가 한 문장에 묶이는 커리어 서사

## 9. 결정 로그 (논의로 확정된 사항)

- [x] AI 분석은 저장 즉시가 아니라 **밤에 일지 쓸 때 하루치 모아 한 번** (비용·품질)
- [x] DB 저장과 AI 분석을 **분리** — DB 먼저, AI 나중 (외부 의존성과 핵심 데이터 생명주기 분리)
- [x] 분석 메뉴를 **A(한 아이) / B(오늘 전체)**로 분리
- [x] 메뉴 A는 **수동 + 월말 자동 스케줄러** 병행
- [x] 메모 입력은 항상 **메모 창 한 곳**에서 (재분석 시에도)
- [x] 배너 판정은 MVP에서 **`메모.생성시각 > 분석.생성시각`**
- [x] 삭제 정책은 **soft delete** (`deleted_at`) — 관찰 기록은 자산
- [x] 일지 중복은 **유니크 키 + 확인 모달**, 재분석 시 덮어쓰기 (버전 관리는 v2)
- [x] 분석 결과도 **DB 저장** — 상담·서류에 재사용
- [x] 스택: **Java + Spring Boot + MySQL + Gradle + Docker**
